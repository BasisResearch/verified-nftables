## The adversarial-verification requirement (applies to EVERY task below)

For each task, after the implementation work, **spawn a SEPARATE, independent, skeptical reviewer
agent** (do not let the implementer grade itself). Its mandate, every time, is to *verify or refute*:

1. **The extracted/shipped code actually runs the VERIFIED term** â€” e.g. the CLI/library invokes the
   extracted `optimize_table_uncond` / `compile_chain` / the new verified passes, NOT a hand-written
   OCaml re-implementation that silently diverges from the proof. Trace the call path from `main`/the
   public API down to the extracted Coq function. Flag any hand-OCaml that duplicates verified logic.
2. **No corners cut** â€” no `Admitted`/`admit`/`Axiom`; every gate green (corpus 2532/2532, validate 28/28,
   parse-test, semtest); headline theorems still axiom-free; no existing theorem weakened or deleted.
3. **Non-vacuity** â€” the new theorems are meaningful (the optimizer actually fires the new merge; the
   pipeline actually transforms input), not trivially true (e.g. `c' = c`, or an unsatisfiable premise).
4. **Honest trust boundary** â€” anything untrusted (renderer, parser, netlink send) is clearly labeled as
   such and differentially tested; claims in README/DEVELOPMENT/LIBRARY are not overstated.

The reviewer returns **PASS/FAIL with specific file:line defects**. On FAIL, the implementer fixes and
the reviewer re-checks before anything is merged. Prefer isolated git worktrees per task; merge only after
PASS + a self-run of all gates.
