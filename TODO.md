## Orientation (current state)

**What this repo is.** A Rocq/Coq-verified nftables compiler + optimizer + parser. The
pipeline, with its trust boundary:

```
.nft text
  └─(UNTRUSTED frontend)─► Syntax.chain        extracted/lexer.mll + parser.mly
                                                + nft_parse.ml + nft_lower.ml
  Syntax.chain (Coq AST)
  └─(VERIFIED)─► optimize_table_uncond ─► optimized Syntax.chain     Optimize_Uncond.v
  └─(VERIFIED)─► compile_chain ─► bytecode (list of instr)           Compile.v
  bytecode
  └─(UNTRUSTED render)─► `nft --debug=netlink` text                  extracted/codec.ml (to_netlink_text)
```

**The verified guarantees (all axiom-free; `Print Assumptions` = "Closed under the global context"):**
- `Correct.compile_chain_correct` — the bytecode VM filters every packet exactly as the DSL semantics says.
- `Optimize_Uncond.optimize_table_uncond_correct` / `…_compile_correct` — the optimizer (and the
  compiled bytecode of the optimized chain) preserves every packet's verdict, **for any input ruleset,
  with NO precondition** (no `rules_clean`, no freshness side-condition). `optimize_table_uncond c` is a
  function of `c` alone; freshness is internal (names minted past `seed_start`).

**Optimizer passes implemented & verified** (compose into `optimize_table`, shipped as `optimize_table_uncond`):
- base `optimize_chain` (dedup of repeated matches, dead-rule elimination, singleton-range→`cmp eq`) — `Optimize.v`
- `setsN`  : value → anonymous set            — `Optimize_Merge.v`
- `vmapN`  : value + differing verdict → vmap  — `Optimize_Vmap.v`  (merge gated on `body_vmap_safe`: no notrack/synproxy)
- `concatN`: two differing selectors → concat  — `Optimize_Concat.v`
- composition + unconditional correctness       — `Optimize_Table.v`, `Optimize_Uncond.v`, seam helpers in `Optimize_Table_Inv.v`

**Data model.** `Semantics.v` `Record set_decls := { sd_sets; sd_vmaps; sd_maps }`. Per-datatype
validity predicates + constructors are in `Nftval.v`. NOTE: `sd_maps` exists but **no pass writes it yet**.

**Tooling layout** (`proof/extracted/`, built with dune):
- `glue.ml` — the CLI (modes: `optimize`, `optsets`, `optdemo`). Entry the executable runs.
- `nftc.ml` / `nftc.mli` — reusable library API (compile/optimize/render + the `.nft` frontend).
- `codec.ml` — bytecode⇄`--debug=netlink` text (render is the untrusted step; differentially tested).
- `lexer.mll` / `parser.mly` / `nft_parse.ml` / `nft_lower.ml` — the `.nft` text frontend (untrusted glue).
- There is **NO actual netlink send** today — the tool only RENDERS `--debug=netlink` text.

**Build & gates** (run from `proof/`, after `eval $(opam env --switch=vst)`):

## The adversarial-verification requirement (applies to EVERY task below)

For each task, after the implementation work, **spawn a SEPARATE, independent, skeptical reviewer
agent** (do not let the implementer grade itself). Its mandate, every time, is to *verify or refute*:

1. **The extracted/shipped code actually runs the VERIFIED term** — e.g. the CLI/library invokes the
   extracted `optimize_table_uncond` / `compile_chain` / the new verified passes, NOT a hand-written
   OCaml re-implementation that silently diverges from the proof. Trace the call path from `main`/the
   public API down to the extracted Coq function. Flag any hand-OCaml that duplicates verified logic.
2. **No corners cut** — no `Admitted`/`admit`/`Axiom`; every gate green (corpus 2532/2532, validate 28/28,
   parse-test, semtest); headline theorems still axiom-free; no existing theorem weakened or deleted.
3. **Non-vacuity** — the new theorems are meaningful (the optimizer actually fires the new merge; the
   pipeline actually transforms input), not trivially true (e.g. `c' = c`, or an unsatisfiable premise).
4. **Honest trust boundary** — anything untrusted (renderer, parser, netlink send) is clearly labeled as
   such and differentially tested; claims in README/DEVELOPMENT/LIBRARY are not overstated.

The reviewer returns **PASS/FAIL with specific file:line defects**. On FAIL, the implementer fixes and
the reviewer re-checks before anything is merged. Prefer isolated git worktrees per task; merge only after
PASS + a self-run of all gates.

## TODO 1 — Close the gap between our optimizer and `nft -o`

**Goal.** Make the optimizer a faithful superset of upstream `nft --optimize`. Two capabilities are
missing today (the three core consolidations — value→set, value+verdict→vmap, 2-field concat — are done).

### 1a. Data / value maps (the `sd_maps` pass)
`nft -o` folds rules whose differing part is a **statement value** (not the verdict) into a map keyed by
a match, e.g.
```
ip daddr A dnat to X
ip daddr B dnat to Y   =>   dnat to ip daddr map { A : X, B : Y }
```
and likewise `meta mark set … map { … }`. The model already has `set_decls.sd_maps`, but **no pass writes
it**. Implement a `mapN` pass (mirror `setsN`/`vmapN` in structure):
- Add the merge logic + a `body_map_safe`-style gate analogous to `body_vmap_safe` (watch for the same
  conntrack/notrack soundness pitfall the vmap pass hit — the merged rule reads its key relative to the
  body; if a ct-dependent key is read after a `notrack` latch it diverges. See `Optimize_Vmap.v`'s
  `body_vmap_safe` + `eval_rules_vmap_mergeN` for the pattern and why the gate is load-bearing).
- Prove an **unconditional** per-pass correctness theorem (`…_uncond`), matching the `Optimize_Uncond.v`
  style (read-freshness via `seed_start`, NO `rules_clean`). Compose it into the pipeline so
  `optimize_table_uncond_correct` still holds with the new stage.
- Wire it into `Extract.v` + `glue.ml` so the shipped optimizer runs it.

### 1b. N-dimensional concatenations (≥3 selectors)
`concatN` is hard-limited to exactly two selectors (`head_value2` matches `MCmp f1 :: MCmp f2 :: rest`).
Generalize to an N-tuple concat (`ip saddr . ip daddr . tcp dport { … }`):
- Generalize `head_value2` → an N-prefix extractor; generalize `concat_merge_pair` and the merge.
- Prove the unconditional per-pass correctness and compose into `optimize_table_uncond`.

**Per-pass discipline (both 1a and 1b):** every new pass needs (i) the function, (ii) an axiom-free
unconditional `_correct` theorem, (iii) extraction wiring so the SHIPPED optimizer uses it, (iv) a
`semtest` battery demonstrating it fires (rules shrink, the map/concat is synthesised) AND that
`eval_chain` agrees with the original on representative packets.

**Adversarial verification for TODO 1:** the reviewer must confirm the new passes' correctness theorems
are unconditional + axiom-free, the merge gates are real (not vacuous — find a notrack/ct case that the
gate must block), the SHIPPED `optimize_table_uncond` actually invokes the new passes (trace
`glue.ml` → extracted term), and corpus/semtest stay green. Cross-check a few outputs against real
`nft -o` (`difftest`-style) to confirm we match upstream behavior where claimed.

## TODO 2 — Tooling: real executable, complete parser, working end-to-end pipeline, netlink send

**Goal.** A user can take a `.nft` file and get verified, optimized rules onto the kernel — with the
trusted core being the extracted verified term and the untrusted edges clearly delimited + tested.

### 2a. Verified optimizer as a first-class executable
Today `glue.ml` is demo-oriented (`optimize`/`optsets`/`optdemo`). Make a clean CLI (e.g. `nftc`):
`nftc optimize FILE.nft` → parse → `optimize_table_uncond` → render; `nftc compile FILE.nft` → bytecode;
sensible flags, stdin/stdout, exit codes. The optimize/compile core MUST be the extracted verified
functions.

### 2b. Complete the parser
`lexer.mll` + `parser.mly` + `nft_lower.ml` cover a large subset but not all of nftables. Inventory what's
unsupported (grammar gaps, statement/expression kinds, address families) and extend coverage. Keep the
`parse-test` + corpus round-trip green as you grow it. The parser is UNTRUSTED glue (it produces
`Syntax.chain` terms the proof reasons about) — so its correctness is established by differential testing
against live `nft`, not by proof; keep that boundary honest.

### 2c. End-to-end pipeline that actually works
Make `.nft → parse → optimize → compile → (render | send)` work as one command, on real rulesets
(e.g. the repo's `router.nft` / `optiplex.nft`). Add an end-to-end test that runs a real `.nft` through
the whole chain and checks the result against live `nft` where possible.

### 2d. Actually send netlink messages
Today the tool only RENDERS `--debug=netlink` text. Add a real sender: open an `AF_NETLINK` /
`NETLINK_NETFILTER` (nfnetlink) socket and submit the compiled ruleset to the kernel (a thin libmnl-style
or hand-rolled nfnetlink batch). **Critical:** the bytes/messages sent MUST be exactly what the verified
`compile_chain` produced — the sender is untrusted transport, it must NOT re-derive or re-encode rules in
a way that bypasses the verified compile. Gate it behind an explicit flag (it mutates kernel state); make
it testable in a netns. Note the two byte-render formats (`codec.ml` corpus big-endian vs the
`--debug=netlink` little-endian 32-bit words) — see memory `two-bytecode-render-formats.md`; don't mix them.

**Adversarial verification for TODO 2:** the reviewer must (1) trace that `nftc`'s compile/optimize path
bottoms out in the extracted verified functions (no shadow OCaml optimizer/compiler); (2) confirm the

## TODO 3 — Ergonomics for per-configuration theorems (readability + tactics)

**Goal.** Make it pleasant to STATE and PROVE properties about a concrete `.nft` ruleset. Today, proofs
like those in `Optiplex_Antispoof.v` / `Router_*.v` are written by hand against the parser-generated
`*_Gen.v` chains, with verbose `cbn`/`unfold`/`vm_compute` boilerplate and ad-hoc wire-byte literals.

- **Readable notations / definitions** for common configuration facts: "rule R drops packet P",
  "chain C accepts P", "field F of P equals V", verdict outcomes, conntrack/fib/nat predicates — so a
  property reads close to the nftables intent rather than raw byte lists. Route literals through the
  central `Nftval.v` constructors (validity-checked datatypes) instead of bare bytes.
- **Custom tactics** (Ltac/Ltac2) that discharge the recurring shapes: evaluating a rule/chain on a
  concrete packet, proving a match fires / does not fire, threading a chain trace rule-by-rule
  (`eval_chain_trace`), reducing `field_value`/`read_payload` to literals. Aim to turn today's
  multi-line `cbn -[…]; unfold …; rewrite …; vm_compute` rituals into one or two named tactics.
- A small **"how to prove things about your ruleset"** guide (in `proof/DEVELOPMENT.md` or a new doc),
  with a worked example: take a `.nft`, `make gen`, state a property with the new notations, prove it with
  the new tactics.
- Keep it ADDITIVE: existing theorems must still compile unchanged (or be re-expressed equivalently,
  byte-equal). This is a library/tactic layer, not a semantics change.

**Adversarial verification for TODO 3:** the reviewer must confirm the new notations/tactics are SOUND —
they don't silently change what's being proved (e.g. a notation that unfolds to a weaker statement, or a
tactic that closes a goal via a hidden `admit`/`Axiom`/over-eager `vm_compute` that masks a real
obligation). Concretely: re-express ≥1 existing hand-proof using the new layer and show it proves the
SAME statement (axiom-free, `Print Assumptions` Closed); attempt to use the tactics to "prove" a
deliberately FALSE configuration property and confirm they fail rather than succeed. All gates green.
